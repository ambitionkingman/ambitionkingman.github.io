案例1 重构项目

问题

新旧数据同步

  1. 前提： 旧系统完全停用或部分城市可用（按城市区分）。
  2. 新旧数据梳理：表结构、字段、数据类型。
  3. 新到旧：消息队列，实时同步。
  4. 旧到新：定时扫表 ,根据增加辅助字段定时进行同步(增加 syn_data_id,syn_data_time) 。

    新旧系统切换

  5. 灰度替换 :（按城市切换，先切换业务量小的城市，稳定运行后，进行大批城市切换）
  6. 旧系统设置 : 切换时，原系统禁用切换新系统城市，防止旧系统进行业务逻辑操作，导致数据错误问题。
  7. 新系统对应app : 切换时，发布新app限定切换城市使用。

    原系统痛点

  8. 数据库结构 : 复杂、冗余、结构混乱、错误数据多、
  9. 数据数量 : 多个核心表百w级数据起始、
  10. 查询sql : 查询缓慢，业务数据直接读取百w级数据表、索引混乱，没有有效的索引结构。
  11. 导出功能 : 导出功能经常导致系统crash，主要是查询sql缓慢，慢查询，系统运行高峰时导出，导致数据库rds 内存飙升，项目无反应，数据库无反应。

    新系统

    架构

    环境组件
  12. 数据库：redis 、mysql

  13. 消息队列：rabbitmq

  14. 项目架构：redis 做内存数据库，mysql 关系型数据库

  15. 业务：（1）创建订单->（2） 派单->（3）确认到达->（4）现场成功/现场失败/待跟进/扑空
  16. 描述：1234同时操作redis 与mysql，创建时订单redis与mysql各存一份，从创建到到达现场数据操作redis与mysql同时操作，4操作完成后从redis中删除，更新mysql数据库做持久化。

技术问题：

概述

当面对较多的并发时，使用redis进行读后写操作，是非常容易出问题的，常常使得程序不具备鲁棒性即健壮性，bug很难稳定复现（得到的值往往跟并发数有关）。

  1. redis 与mysql 数据一致性问题：（1）数据库事务；（2）redis 事务；解决办法：1.数据库事务使用注解@Transaction，2.redis可使用watch multi exce 等命令(项目中涉及多条数据更新的地方很少)。todo
  2. redis 并发问题：分布式/并发 同一条数据锁问题
  3. 分布式问题：（1）session 共享，（2）websocket共享，（3）netty server ；解决办法：12可以使用spring session
  4. 单一职责原则或业务隔离：（1）redis中心跳主体只包括心跳数据，不包括其他业务数据；（2）redis 中及时雨人员数据只包括人员数据，不包含其他业务数据。（3）redis中订单业务数据只包括订单业务数据。
  5. 系统容错：统计纠正接口
  6. 原改派问题处理解决：解决办法：1.先读后写 （改为只写不读或加锁 肯定不行的），2.使用redis watch 命令，待操作测试。

建议

设计问题：

架构
  1. 业务逻辑一定要屡清楚：不然代码没法写，逻辑不清楚，只能边写边发现（太难受了），而且有些问题也不容易发现。（开业务讨论的时候最好参与的人都能够参加，时间后面都能够找的回来）；
  2. 架构中相关组件：特性一定要调研清楚，否则后面出现问题，很难处理（讲道理开发之前会有一段时间做调研的）。
  3. 业务与数据表设计一定要相合才行；不能存在冲突，如：某个ziduan字段字段在a表中有，但是某些特殊数据不会再a表写记录，然后ziduan，还有逻辑处理，查询更新等操作，根本没字段存，你让我怎么操作。（尴尬的一匹），后面改表实在是浪费开发时间（动辄一到两天，一起商量设计也就半天左右）。
    数据库
  4. 数据库结构一定要复核业务，而且粒度尽可能细化，如状态、类型、，一个表20-30字段，(我觉得没啥问题)。（只要保证大部分字段使用频率较高且大部分生命周期都有值的。频率低的可以单独抽成表，或者与部分查询统计有关的可以存在冗余，不如大部分查询，如关联的公司，门店，人，都可以冗余相对应的名称）
  5. 数据库的注意事项特殊标明：如不能扫表，不能做大数据量导入，大数据量查询，不能锁表等。
  6. 数据库事务：常见处理方法 注解
  7. 数据库锁问题：没有并发写入验证（应该不需要行锁，表锁）

    业务问题：

    文档
  8. 文档一定要出、出、出：对于一些常见的，如：（1）页面输入框的类型，长度，上下限，默认值；（2）多个输入框之间的关系，如时间输入框，开始时间一般都不会大于结束时间。这种存在起始小于终止的。可以做成一个规范，开发时规范即可。
  9. 业务涉及的数据，中不能空的字段，范围等，提前敲定（否则后期改代码太耽误事了，设计好的话直接就限定）；
  10. 页面：所有的查询要有条件，如时间，是否需要默认值，开始时间不可以大于结束时间，时间格式，日期，日期时间，时间，文本框输入长度的限制，
  11. toast 文本，可以创建规范。
  12. 列表：都哪些是模糊搜索，按什么排序，是否分页，查询列标题；是否关联操作。
  13. 日志：最好有统一的日志规范；操作时间，操作人，描述，备注，
  14. 一再原有功能基础之上抽离出设计来。否则太丑了，太功能化了。
  15. 各个业务跳转等，要有详细的流程图
  16. 文档的及时性，先于开发才行。
    业务逻辑
  17. 业务一定要提前共享出来：把产品的所思所想同步给开发

    Watch命令
    大家可能知道redis提供了基于incr命令来操作一个整数型数值的原子递增，那么我们假设如果redis没有这个incr命令，我们该怎么实现这个incr的操作呢？那么我们下面的正主watch就要上场了。
    如何使用watch命令
    正常情况下我们想要对一个整形数值做修改是这么做的(伪代码实现)：
    val = GET mykey
    val = val + 1
    SET mykey $val
    但是上述的代码会出现一个问题,因为上面吧正常的一个incr(原子递增操作)分为了两部分,那么在多线程(分布式)环境中，这个操作就有可能不再具有 原子性 了。
    研究过java的juc包的人应该都知道cas，那么redis也提供了这样的一个机制，就是利用watch命令来实现的。
    WATCH命令可以监控一个或多个键，一旦其中有一个键被修改（或删除），之后的事务就不会执行。监控一直持续到EXEC命令（事务中的命令是在EXEC之后才执行的，所以在MULTI命令后可以修改WATCH监控的键值）

案例n 高并发支持

原系统

  1. 环境
    同一请求并发高峰1000左右，单表数据量1千万左右。

    解决方案

  2. 环境
    阿里的三台服务器：东北，华北，华南。
  3. 要求
    读写分离
    访问分流
  4. 概述: 数据库的读写，使用数据分析主从架构，主数据库负责写操作，从数据库负责读操作。
  5. 解决方案：根据用户的ip进行访问分流。分区服务器压力，根据ip地址库，获取地址计算距离。或者调用第三方接口获取地址，访问最近服务器
  6. 缓存层处理
    • 概述：使用redis 或其他缓存工具进行缓存，将读取周期长的sql缓存起来。
  • 例如：可以取sql的hash做key，把查询结果做vlue缓存。
  • 优点：对程序员来说是无感的，他们开发时是不会知道有这么一层的，不需要改动任何逻辑。
  1. 其他解决办法，或优化办法。
  • 比如会有一个代码集中部署方案。
  • 比如会有一个平台 集中观测各个服务器运行状态。
  • 这两个方案用到了一个socket 框架。
  • 执行Linux 命令用websocket 送到前台。
  • 代码自动部署用svn钩子 用socket 通知所有服务器更新代码，返回执行结果给用户。

就是先用Update之行，把并发问题抛给数据库去解决。之行成功后存入消息列队！后台处理接下来的问题，最后从消息队列里面取得执行结果嘛？然后返回，减少执行时间。
